El flujo de una orden, de principio a fin

Indigo Decors fabrica y decora puertas para dealers (Lock Tight, USA Windows, Web Indigo, etc.). Una orden entra por dealer, pasa por 13 etapas, y termina cobrada + el pintor e instalador pagados.

1. Captura

Office crea la orden con cliente, dealer, diseño y dimensiones

2. Medición

Javier va al sitio, confirma medidas. SQF se calcula

3. Digitalización

Pedro carga el SQF por pieza desde su CorelDraw

4. CNC

Ramon corta las piezas en el router

5. Pintura

Mario pinta. Al terminar se le genera la liquidación automática (SQF × $8)

6. Instalación

Carlos instala, sube foto + firma. Genera payout $35/puerta

7. Facturación

Beatriz registra el cobro al dealer y la marca como pagada

8. Liquidación

Beatriz consolida y le paga a Mario y Carlos

💡 Idea clave: El sistema le muestra a cada persona solo lo que tiene que hacer ese día. Mario no ve dashboards ni facturación, solo ve sus órdenes a pintar. Beatriz no toca el canvas de instalación, ve cash flow + facturación pendiente.

02 Manager · Office

Dashboard

La vista 360° del taller. Le dice a Majela: ¿cuánto hay activo, qué está atascado, cuánto vamos a cobrar este mes?

Qué muestra

  • ●5 KPI cards: Active Orders, Installations Pending, CNC Queue, Painting, Revenue this month
  • ●Revenue last 6 months: chart de barras con cobros reales por mes
  • ●Production Funnel: barras horizontales por etapa
  • ●Orders by Company: pie chart con % y revenue por dealer
  • ●Production Board: las 6 etapas principales con la card más vieja de cada una (click → Kanban)
  • ●Today's Routes: instalaciones de hoy
  • ●Production Capacity: barras de % utilización (CNC, Painting, Install) basadas en Settings
  • ●Health: orders overdue, días promedio en etapa, instaladas/pendientes esta semana

Cómo probarlo

  1. Login como Manager o Office
  2. Verificá que los KPI no tengan placeholders inventados (no "+12% vs last week")
  3. El chart de Revenue debe mostrar barras reales (junio destacado en azul Indigo)
  4. Click en cualquier card de Production Board → abre Kanban
  5. Click en "View all" del pie chart → abre Orders
  6. Click en "Open planner →" en Today's Routes → abre Route Planner
  7. Las barras de capacity están enlazadas a Settings (cambiá un valor allá y refrescá Dashboard)

04 Todos · acciones por rol

Order detail

La cara de una orden individual. Ficha técnica, cliente, piezas, asignaciones, chatter y el botón de avanzar de etapa.

Qué muestra

  • ●Header: número de orden, badge de etapa actual + botones Print y wizard de la etapa actual
  • ●Imagen + ficha técnica: door type, color, medidas, brand, glass, privacy
  • ●Client Information: reference, nombre, dirección, teléfono, email
  • ●Note: notas + total SQF
  • ●Pieces: tabla con cada pieza (design, type, color, brand, glass, privacy, W×H, qty, SQF)
  • ●Recent Activity: chatter de Odoo (cada cambio de etapa, asignación, foto, etc.)
  • ●Order Progress (sidebar): las 13 etapas con check verde / círculo azul (actual) / gris
  • ●Order Summary: door type, color, glass, SQF, status, total
  • ●Assigned Contractors: painter y installer(s) actuales + botón Edit (Manager/Office)

Cómo probarlo

  1. Login como Manager
  2. Orders → click en IND/2026/00010 (Sebastian Cortes, Measurement Pending)
  3. Verificá la sidebar de progreso: las primeras etapas con check, "Measurement Pending" como actual
  4. Click en Enter measurements (botón verde arriba a la derecha) → abre wizard
  5. Cancelá el wizard, scroll hasta el final → vas a ver el chatter con cada cambio
  6. En Assigned Contractors → click Edit → cambiar painter / installer → Save
  7. El chatter debe mostrar "Painter / installer assignment updated from Next panel"
  8. Click en Print → descarga el Order Card PDF
⚠ Importante: Si la orden no tiene painter / installer asignado y vas a avanzar desde Painting → Ready for Installation, el wizard te bloquea con un banner ámbar y un Save deshabilitado. Esto evita generar órdenes sin payout. Asigna primero desde la card de Assigned Contractors.

📄 Documentos PDF que genera el sistema

Todos se generan automáticamente con QWeb desde Odoo, accesibles con un click en la app.

1. Order Card (ficha de orden)

Se imprime y se le entrega al cliente. Disponible desde Orders, Order detail, Stage screens.

2. Order Label (etiqueta del diseñador, 12.7 × 57.15 mm)

Se pega detrás de cada pieza cortada. Trae QR con el order.name + medidas + PRIV + parts + diseño.

3. Painter Sheet (Hoja del pintor)

Company / Order / Cliente / Color / Door Type / SQF / Price / Total. Imprimís desde /paint.

4. Installation Addresses

Lista de paradas con dirección, teléfono, instaladores. Imprimís desde /route-planner.

5. Payout Receipt

Recibo de pago al contratista. Click en 🖨 al lado de cada payout en /billing.

🔄 Relación con Odoo

La app Next no tiene base de datos propia. Es un cliente Odoo con UI a medida. 

Usuario hace algo en la app Next
  ↓ /api/orders/8/assign { painter_id: 33 }
Next BFF (proxy)
  ↓ JSON-RPC POST /web/dataset/call_kw
  model: indigo.order, method: write, args: [[8], { painter_id: 33 }]
  Cookie: session_id=<session del usuario logueado>
Odoo backend
  → record rules verifican permisos del usuario
  → escribe en Postgres
  → trigger en write() ejecuta _create_painter_payout / _create_installer_payouts
  → message_post() al chatter
  → notifica a usuarios suscritos al record
  • ✓ Todo lo que ves en la app vive en Odoo
  • ✓ Cualquier cambio se ve también en el backend de Odoo (incluyendo el kanban nativo)
  • ✓ Las record rules y los permisos de Odoo aplican igual: un Painter no puede ver órdenes que no le pertenecen
  • ✓ Los reportes PDF los renderiza Odoo con QWeb
  • ✓ El chatter, attachments, activities — todo de Odoo
¿Por qué entonces hacer la app Next?

Para darle a cada rol una UX hecha a su medida: el pintor abre y ve solo lo que tiene que pintar. El instalador abre desde el celular y tiene un canvas táctil para la firma del cliente. Beatriz tiene un Cash Flow al primer vistazo. El backend de Odoo es potente pero pensado para administradores — no para operativos.
